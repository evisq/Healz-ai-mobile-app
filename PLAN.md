# План выполнения тестового задания Healz Mobile

## 1. Итоговый технический выбор

### Стек

- Expo SDK 56 + React Native + TypeScript.
- `react-native-webview` для публичного `https://app.healz.ai`.
- `expo-share-intent` 7.x для получения PDF и изображений из Android Share Sheet.
- Expo Dev Client, а не Expo Go: Share Intent и PDF-компрессор требуют нативного кода.
- EAS Build для подписанного устанавливаемого APK.

### Почему Expo, а не Capacitor

- Исходников веб-продукта нет, поэтому преимущества Capacitor для упаковки существующего web-кода почти не используются.
- Основная часть задания — нативная: входящий Share Intent, локальная обработка файлов, Android lifecycle и связка файла с WebView.
- Expo даёт единый React Native-код для iOS/Android, готовый config-plugin для Share Intent и простой выпуск APK через EAS.
- iOS-сборка сейчас не нужна, но выбранные интерфейсы и основной JS-код остаются кроссплатформенными.

## 2. Что уже установлено и выяснено

Этап 0 выполнен 25 июня 2026 года. Создан Expo SDK 56 проект и подготовлен
GitHub-репозиторий.

В окружении уже есть:

- Node.js 24 LTS и npm 11;
- Temurin JDK 17;
- Android SDK Platform и Build Tools 36/37;
- Android Platform Tools и `adb`;
- Expo CLI через `npx` и EAS CLI;
- Git;
- VS Code;
- Google Chrome;
- авторизованный GitHub CLI;
- `scrcpy`;
- FFmpeg;
- `pdfinfo`.

Android Studio и эмулятор сознательно не установлены: разработка ведётся в
VS Code, а проверка Share Intent будет выполняться на физическом телефоне.
Остаются два ручных действия: `eas login` и авторизация подключённого телефона
для USB debugging.

По публичному клиентскому коду Healz:

- email-вход использует шестизначный OTP;
- auth API находится на том же origin: `/api/auth`;
- после входа используется маршрут `/app/chat`;
- сессия восстанавливается клиентом при запуске;
- мобильная верстка и загрузка вложений на сайте уже предусмотрены.

Это означает, что сначала следует использовать штатное cookie-хранилище Android WebView. Сохранять auth-cookie вручную в AsyncStorage нельзя без необходимости: это ухудшит безопасность.

## 3. Пользовательский сценарий MVP

### Обычный запуск

1. Пользователь открывает приложение.
2. Показывается брендированный splash.
3. Открывается `app.healz.ai` внутри WebView.
4. Пользователь входит по email и OTP.
5. После закрытия, force-stop и повторного запуска сессия остаётся активной.

### Отправка документа

1. Пользователь открывает PDF или фото в Gmail, Files, Google Photos или другом приложении.
2. Нажимает «Поделиться» и выбирает Healz.
3. Healz получает один или несколько файлов через Android Share Intent.
4. Приложение копирует их во внутренний cache и проверяет тип/размер.
5. Файл локально и без внешних серверов аккуратно сжимается.
6. Пользователь видит понятный итог, например: `8,4 МБ → 2,3 МБ`, имя и тип файла.
7. По кнопке «Добавить в Healz» приложение открывает чат и передаёт файл в штатный upload сайта.
8. После успешной передачи временная копия удаляется.

### Почему такой UX

- Он начинается там, где документы уже находятся, и не заставляет сначала открывать Healz.
- Системный Share Sheet одинаково привычен для почты, файлового менеджера и галереи.
- Экран подготовки объясняет небольшую задержку и показывает пользователю пользу сжатия.
- Файл не отправляется автоматически: пользователь подтверждает действие и видит, что именно будет загружено.

## 5. Главный технический риск: передача файла в чужой WebView

У нас нет исходников Healz и нельзя добавить официальный JavaScript bridge на стороне сайта. Поэтому этот сценарий нужно проверить первым отдельным spike-прототипом.

### Вариант A — основной

1. Открыть `/app/chat`.
2. Найти штатный `input[type="file"]`; если он создаётся лениво — сначала нажать кнопку вложения.
3. Передать обработанный файл в WebView.
4. Создать `File`/`DataTransfer` в контексте страницы.
5. Назначить `input.files` и отправить события `input`/`change`.

Плюсы:

- мало нативного кода;
- основной механизм одинаков для iOS и Android;
- сайт продолжает сам выполнять загрузку и обработку.

Минусы:

- DOM публичного сайта может измениться;
- большие base64-пакеты могут давить на память и Android WebView bridge.

Защита от хрупкости:

- искать generic `input[type=file]`, а не CSS-классы;
- иметь несколько стабильных fallback-селекторов по `aria-label`;
- ограничить размер файла после подготовки;
- передавать файлы последовательно;
- показывать понятную ошибку, а не терять документ;
- покрыть скрипт bridge unit-тестами на нескольких DOM-вариантах.

### Вариант B — fallback, если A не проходит spike

Сделать небольшой Android-мост для `WebChromeClient.onShowFileChooser`:

- приложение хранит URI подготовленного файла;
- программно нажимает штатную кнопку вложения;
- нативный file chooser возвращает WebView этот URI как будто файл выбрал пользователь.

Это сложнее и Android-специфично, но устойчивее для больших файлов и не требует передачи base64 через JS.

### Критерий завершения spike

На физическом Android-телефоне PDF и JPEG, полученные из другого приложения, появляются в форме чата Healz и реально загружаются. Пока это не доказано, не следует тратить время на polish.

Для проверки потребуется тестовый Healz-аккаунт с доступом к email.

## 6. Подготовка файлов

### Общие правила

- Только локальная обработка в app sandbox.
- Никаких сторонних API компрессии: речь идёт о медицинских документах.
- Исходник не удаляется.
- Подготовленная копия используется, только если она валидна и действительно меньше.
- При ошибке пользователь может отправить оригинал.
- Не писать содержимое, base64 и чувствительные имена файлов в логи.

### Изображения

Инструмент: `expo-image-manipulator`.

Алгоритм:

1. Прочитать MIME, размеры и EXIF-ориентацию.
2. Не увеличивать небольшие изображения.
3. Для крупных документов уменьшить длинную сторону примерно до 2400–3000 px.
4. JPEG сохранять с качеством около 0,85.
5. PNG/графику сравнивать с JPEG-версией и брать меньший валидный файл.
6. Если экономия меньше примерно 10%, оставить оригинал.
7. Проверить декодирование результата и минимальные размеры.

2400+ px по длинной стороне обычно сохраняют мелкий текст анализов читаемым на приближении и дают заметное уменьшение фотографий с камеры.

### PDF

Предпочтительный MVP: небольшой локальный Expo-модуль на Kotlin с `pdfbox-android`.

Алгоритм:

1. Открыть PDF локально.
2. Не растрировать страницы целиком.
3. Сохранить текстовые и векторные слои.
4. Пережимать только встроенные растровые изображения с умеренным JPEG quality.
5. Сохранить новый PDF.
6. Повторно открыть результат и проверить количество страниц.
7. Использовать результат только если он меньше исходника.

Это безопаснее для медицинских документов, чем превращать каждую страницу в размытую картинку. Для зашифрованных, повреждённых или неподдерживаемых PDF применяется оригинал с пояснением пользователю.

В качестве проверенного референса алгоритма можно использовать Apache-2.0 проект ImageToolbox, который пережимает embedded images через `pdfbox-android`, сохраняя остальное содержимое PDF.

## 7. WebView

### Базовые настройки

- URL только `https://app.healz.ai`.
- DOM storage и JavaScript включены.
- Не использовать incognito/ephemeral WebView.
- Разрешить cookie и third-party cookie там, где это требуется.
- Вызвать flush cookie-хранилища после успешного входа только если тест покажет проблему.
- Не хранить пароль или OTP.

### Навигация

- Android Back:
  - если WebView может вернуться назад — `goBack()`;
  - иначе — штатное поведение выхода из приложения.
- Ссылки внутри `app.healz.ai` остаются в WebView.
- `mailto:`, `tel:`, карты и внешние HTTPS-домены открываются системными приложениями/браузером.
- `target="_blank"` перехватывается и обрабатывается явно.
- Блокируются небезопасные схемы и неожиданные переходы.

### Состояния

- splash до готовности приложения;
- индикатор первой загрузки;
- экран ошибки сети с Retry;
- понятная ошибка импорта;
- очередь файлов при нескольких вложениях.

## 9. Стратегия проверки

### Автоматические тесты

- URL policy: внутренние и внешние ссылки.
- Логика Android Back.
- MIME/type validation.
- Выбор оригинала или compressed copy.
- Очистка временных файлов.
- Генератор WebView injection script.
- Ошибки и fallback-состояния.

### Файловый test corpus

- JPEG-фото анализа с мелким текстом.
- PNG-скриншот.
- PDF с selectable text.
- Отсканированный многостраничный PDF.
- Уже хорошо сжатый PDF.
- Большой PDF.
- Повреждённый/зашифрованный PDF.
- Файл с Unicode/пробелами в имени.
- Файл с неверным расширением, но корректным MIME.

### Ручная матрица на физическом Android

- Первый вход по OTP.
- Закрытие и повторный запуск.
- Force-stop и запуск.
- Перезагрузка телефона и запуск.
- Share из Gmail.
- Share из Files.
- Share из Google Photos.
- Один PDF, одна картинка и несколько картинок.
- Реальная загрузка вложения в чат.
- Back navigation.
- Внешние ссылки.
- Нет сети во время запуска и во время импорта.
- Отмена импорта.
- Большой файл и нехватка памяти.

### Проверка качества компрессии

- Сравнить размер до/после.
- PDF открывается, число страниц совпадает.
- Текстовый слой PDF не исчез.
- Увеличить мелкий текст до 200–300% и проверить читаемость.
- Не принимать compressed output, если он больше исходника.

## 11. Какие знания нужны

Глубоко знать конкретный язык не требуется, но для контроля AI-агента важно понимать:

- базовый TypeScript и React: компоненты, hooks, async state;
- жизненный цикл WebView и cookie;
- Android Share Intent, MIME types и `content://` URI;
- различие app cache и публичного хранилища;
- основы Kotlin/Gradle для небольшого нативного модуля;
- APK signing и build profiles;
- чтение `adb logcat` и Chrome WebView Inspector;
- основы безопасной работы с медицинскими файлами.

Главный навык здесь — не писать всё вручную, а разбивать работу на проверяемые вертикальные сценарии и не принимать результат агента без запуска на реальном устройстве.

## 12. Как вести AI-агента

Работать короткими проверяемыми итерациями:

1. Дать агенту ТЗ и зафиксированный acceptance checklist.
2. Просить сначала исследовать неизвестные, затем предложить минимальную реализацию.
3. На каждый этап задавать один конкретный outcome.
4. После изменения требовать:
   - список затронутых файлов;
   - объяснение решения;
   - команды проверки;
   - известные ограничения.
5. Самостоятельно запускать тесты и проверять на телефоне.
6. Ошибки возвращать агенту с полным logcat/stack trace, а не пересказом.
7. Делать небольшие осмысленные commits после рабочих вертикальных срезов.
8. Вести `docs/agent-log.md`: задача, решение агента, как проверено, что пришлось исправить.

Для видео ценнее показать два конкретных примера:

- агент предложил решение, которое не сработало на реальном WebView, и оно было заменено после проверки;
- компрессия уменьшала файл, но портила читаемость, поэтому были изменены параметры/алгоритм.

## 13. Приоритеты и разумные сокращения

Порядок ценности для пользователя:

1. Надёжный вход и сохранение сессии.
2. Share из другого приложения и реальная загрузка в чат.
3. Безопасная локальная подготовка файлов.
4. Back/external links.
5. Иконка, splash и polish.

Если времени мало, допустимо сократить:

- multiple-file импорт до одного файла за раз;
- сложные анимации;
- iOS-специфичную доводку;
- автоматизированный end-to-end тест OTP;
- продвинутый PDF downsampling.

Нельзя вырезать:

- установочный APK;
- проверку сохранения сессии;
- хотя бы один рабочий путь PDF и изображения из Share Sheet в чат;
- локальную подготовку файла;
- проверку читаемости;
- обработку ошибок без потери исходника.

## 15. Оценка времени

При готовом окружении и тестовом аккаунте:

- WebView shell: 0,5–1 день.
- Share/upload spike: 0,5–1,5 дня.
- Компрессия и UX: 1–1,5 дня.
- Тесты, polish, APK, README и видео: 1 день.

Реалистичный объём: 3–5 сфокусированных рабочих дней. Самая большая неопределённость — программная передача файла в штатный uploader Healz без изменений на стороне сайта.
